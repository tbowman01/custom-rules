
# Implementation Plan — Custom Rules CLI

Generated by Copilot

This document translates the PRD into concrete architecture, phases, and technical decisions. It reflects the current codebase (TypeScript CLI + template/config scaffolding) and outlines next steps toward a production‑ready rules engine.

## 1. Scope

Core deliverable: a Node.js CLI (`custom-rules`) that can:
- Initialize baseline configuration + profiles + IDE integration files (VSCode, JetBrains, Vim).
- Manage profiles (list/create/delete/show).
- Validate configuration integrity (YAML + structural).
- Execute rules (currently simulated) and output reports (json|yaml|html).
- Provide an extensible path for real rule execution (future).

Out of scope (current phase): real static analysis, remote fetching, interactive wizards (beyond stubs), plugin marketplace.

## 2. Assumptions
- Node.js ≥ 18 environment (ESM capable).
- Users run commands in project root; no monorepo multi-root awareness yet.
- File system operations are local (no remote FS abstraction).
- Rules will evolve to pluggable modules (likely under `src/rules/` or external packages).
- YAML is the canonical persisted config; in‑memory normalized object used at runtime.

## 4. Config Loading Strategy
1. Start with in-memory defaults (profile name as config name).
2. Search precedence (first hit wins and overrides defaults):
   1. Explicit `--config` path
   2. Project root: `custom-rules.yaml|yml|.custom-rules.yaml|yml`
   3. Profile overlays: `profiles/<profile>.yaml|yml` (if present)
   4. User home: `~/.config/custom-rules/config.yaml`, `~/.custom-rules.yaml`
3. Merge model: shallow override; future enhancement: deep merge + schema normalization.
4. Access via `ConfigManager.get()` + mutation via `set()` (mutable; may become immutable snapshot later).

## 8. Error Handling & Exit Codes
- Commander `exitOverride()` used; custom try/catch in `main()`.
- Unhandled Promise + uncaught exception handlers terminate with code 1.
- `run`: exits 1 if any rule failed.
- `validate`: exits 1 if structural errors (warnings tolerated).
- Future: structured error class hierarchy (ConfigError, RuleLoadError, RuleRuntimeError).

## 9. Logging Strategy

## 14. Performance & Concurrency

Planned: worker pool (config.settings.max_workers) using Promise pool; optional cancellation token; per‑rule timing metrics.

|------|--------|------------|

| Plugin execution security | Code injection | Sandbox / allowlist + docs |

## 20. Appendix: Minimal Rule Engine Sketch
```.ts
async function runRules(rules: RuleDefinition[], ctx: RuleContext) {
  const limit = pLimit(ctx.config.settings.max_workers || 4);
  const executions = rules.map(r => limit(() => r.run(ctx)));
  return Promise.all(executions);
}
```
Adds concurrency with backpressure; integrate after Phase 2.
